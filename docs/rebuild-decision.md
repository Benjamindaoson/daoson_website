# 重建决策

结论：**推荐方案 D**

> 先用 V0 产出视觉方向和关键组件，再把设计适配回现有 Jekyll 内容系统。  
> 不建议直接迁移到 Next.js。

## 为什么不是 A

方案 A 是“继续在当前项目上小修小补”。它上线最快，但不够解决当前的核心问题：

- 首屏不够求职导向
- 项目案例不够结构化
- 简历入口缺失
- 首页更像研究日志，不像作品集入口

A 能保住内容，但很难把招聘展示拉到位。

## 为什么不是 B

方案 B 是“保留 Jekyll，但重构当前网站结构和视觉系统”。  
它其实是一个很好的方向，但如果没有一个明确的视觉原型和组件方向，重构容易变成漫长的手工试错。

所以 B 是正确方向之一，**但 D 比 B 更高效**：先让 V0 帮你收敛视觉语言，再把它落回 Jekyll。

## 为什么不是 C

方案 C 是“V0 重新生成 Next.js + Tailwind 网站并迁移内容”。  
这条路的代价太高：

- 要迁移 Markdown、Front Matter、图片、页面模板、SEO、RSS、sitemap、标签、归档、URL、重定向
- Jekyll 已经有的双语、collections、自定义插件、Pagefind、OG 图链路都要重新实现
- SEO 和 URL 风险高
- 迁移工作量大，但收益不一定比当前架构高

对于一个内容为主、长期维护的个人品牌站，这不是最省事的路。

## 为什么选 D

当前站点已经有这些可复用资产：

- 内容体系：posts / notes / til / projects / elsewhere / teaching
- 双语系统：`lang`、`translation_key`、`hreflang`
- 搜索：Pagefind
- SEO：sitemap、robots、canonical、OG、RSS
- 工程化：自定义 plugin、GitHub Actions、内容 schema

所以最优策略不是推倒，而是：

1. 用 V0 收敛视觉方向
2. 用 Jekyll 继续承载内容与结构化数据
3. 先做能用于求职的最小版本
4. 再逐步补项目案例和内容深度

## 推荐方案

**推荐方案：D**  
**推荐指数：8 / 10**  
**预计工作量：中等，1-2 周可做出求职版，3-4 周可做得更完整**  
**预计主要收益：保留现有内容资产，同时把视觉和招聘展示抬起来**  
**最大风险：如果项目案例内容不补齐，页面再漂亮也还是“像博客”**

## 当前网站是否值得继续使用

值得。  
它不是要被丢掉的旧站，而是一个已经有内容体系、SEO 基础、双语机制和工程化能力的底座。

## 当前最有价值的资产

1. Jekyll 内容系统
2. 双语 + translation key 机制
3. 自定义插件：wikilink / backlinks / OG 图
4. Pagefind 搜索
5. 已有文章、笔记、TIL 和教学索引
6. GitHub Actions 部署链路

## 当前最严重的五个问题

1. 没有简历页
2. 没有结构化项目案例页
3. 首页没有把目标岗位说得足够直接
4. 真实项目只有 1 个，且还是 build log 级别，不是案例级别
5. 站点更像数字花园，而不是招聘作品集

## 这些问题能否在当前架构内解决

能，而且应该在当前架构内解决。

Jekyll 已经足够支撑：

- 项目案例结构化管理
- 双语内容
- 技术文章
- 标签和归档
- 简历页
- SEO / sitemap / RSS
- 响应式设计
- 自动部署
- 未来在线 Demo 链接
- 未来 AI Chatbot 入口

## 是否有必要迁移到 Next.js

**没有必要。**

除非你准备同时重做：

- 内容体系
- 页面结构
- 组件系统
- 部署流程
- SEO 路线

否则 Next.js 只是增加迁移成本，不会自动让你更像高级 AI 工程师。

## 是否应该使用 V0

应该，但只用在正确的地方。

### 适合交给 V0 的内容

- 首页首屏视觉方向
- 项目卡片 / 案例卡片
- About / Resume / Contact 的布局草图
- 作品集信息架构的不同方案

### 不要依赖 V0 的内容

- Markdown 内容迁移
- Jekyll collections 设计
- bilingual 逻辑
- SEO / canonical / sitemap / RSS
- Pagefind / OG 图 / backlinks / wikilink 这类站点能力

### 仍需 Codex / 人工完成的内容

- 最终文案
- 项目案例事实
- 职责、难点、结果、证据
- 简历真实信息
- URL 保持与重定向策略
- Jekyll 端的落地和调试

## V0 生成后还需要什么工程化工作

1. 组件落地到 Jekyll
2. 响应式补齐
3. 可访问性修正
4. SEO 元信息补齐
5. 文章 / 项目数据对接
6. 真实内容替换掉 demo 文案
7. 浏览器验证
