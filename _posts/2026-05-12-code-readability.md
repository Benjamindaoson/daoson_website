为什么有的代码读起来像散文，有的却像加密电报？

写代码这么多年，我越来越觉得，**代码的可读性，本质上是「认知负担」的问题**——读者每读一行，要在脑子里维护多少状态？

## 命名是第一道关

```javascript
// 不好
function f(x, y) { return x.filter(i => i.s === y); }

// 好
function findUsersByStatus(users, status) {
  return users.filter(user => user.status === status);
}
```

短变量名节省了打字时间，但偷走了未来读者的时间——而后者通常是几十倍的代价。

## 抽象层级要一致

一个函数里，如果一会儿在做高层级的业务编排，一会儿又在做底层的字节操作，
读者就得不停地切换视角，非常累。

## 写给未来的自己

我的一个习惯：**在每个稍复杂的函数前，写一句话告诉未来的自己「这是什么、为什么」**。

不是注释每一行（那是糟糕的注释），而是告诉读者「上下文」。

```python
# 因为后端在 v2 之前会返回字符串型的 ID，
# 这里强制转 int 以保证前后端类型一致
user_id = int(payload.get('id'))
```

这种注释，写代码的人自己半年后回来看都会感谢自己。

## 小结

可读性不是「好看」，而是**「下一个人接手时的痛苦最小化」**。
而那个「下一个人」，大概率就是几个月后的你自己。
